## Requirements
### Requirement: Feature suppression SHALL bypass feature execution without deleting authored history
The system SHALL allow a committed feature to be marked suppressed so geometry generation skips that feature while preserving its authored feature record, feature order entry, document-history row, label, definition, and edit target.

#### Scenario: Suppressed feature is skipped during rebuild
- **WHEN** a committed feature is marked suppressed
- **AND** the document cursor includes that feature in the applied history range
- **THEN** geometry generation does not execute that feature's kernel operation
- **AND** the rebuilt snapshot keeps the feature row with `suppressed: true`
- **AND** the rebuilt snapshot does not expose bodies, constructions, render records, references, or produced targets generated by that suppressed feature

#### Scenario: Suppressed feature remains editable
- **WHEN** a committed feature is marked suppressed
- **THEN** the feature remains in authored feature order and document history
- **AND** the user can still reopen or edit that feature through the committed feature edit flow

#### Scenario: Suppression does not move the document cursor
- **WHEN** a feature is suppressed or unsuppressed
- **THEN** the document cursor remains at the same authored history position unless that exact cursor target no longer exists for another reason

### Requirement: Suppression state SHALL be mutated through a dedicated modeling operation
The modeling boundary SHALL expose a dedicated mutation for setting the suppression state of one committed feature.

#### Scenario: Suppress existing feature
- **WHEN** a caller requests suppression for an existing committed feature using the current revision and repository basis
- **THEN** the mutation is accepted
- **AND** the returned rebuild result reflects the document regenerated with that feature bypassed
- **AND** changed targets include the feature target and any durable targets invalidated by the rebuild

#### Scenario: Unsuppress existing feature
- **WHEN** a caller requests unsuppression for an existing committed feature using the current revision and repository basis
- **THEN** the mutation is accepted
- **AND** the feature participates in rebuild at its authored history position
- **AND** generated geometry and diagnostics are recomputed from the full applied history state

#### Scenario: Suppression no-op
- **WHEN** a caller requests the suppression state that the feature already has
- **THEN** the mutation returns a skipped rebuild result with reason `noOp`
- **AND** no authored document revision is committed

#### Scenario: Missing feature is rejected
- **WHEN** a caller requests suppression for a feature id that is not present in the authored document
- **THEN** the mutation is rejected with a machine-readable missing-feature diagnostic
- **AND** the authored document remains unchanged

### Requirement: Suppression fallout SHALL be reported as repairable rebuild diagnostics
The system SHALL treat invalid downstream references caused by a suppressed upstream feature as repairable rebuild failures on the downstream authored features.

#### Scenario: Downstream feature references suppressed output
- **WHEN** feature `b` references geometry produced by feature `a`
- **AND** feature `a` is marked suppressed
- **AND** feature `b` remains unsuppressed and inside the applied cursor range
- **THEN** feature `b` remains in authored history
- **AND** geometry generation reports a repairable diagnostic for feature `b`
- **AND** the system does not delete, rewrite, or automatically suppress feature `b`

#### Scenario: Unsuppress repairs downstream references
- **WHEN** a suppressed upstream feature is unsuppressed
- **AND** its downstream references are otherwise valid
- **THEN** the next rebuild resolves the downstream references normally
- **AND** diagnostics caused only by the suppression fallout are cleared
