## Context

Feature editor forms are generated by feature authoring definitions and rendered by the generic `FeatureInspector`. Advanced features such as sweep, loft, and thicken already expose `targetBody` reference collections only for non-create operation intents. Basic boolean-capable features such as extrude, revolve, and shell preserve `operation` plus `booleanScope`, but their forms do not consistently expose a target selector when users choose join, cut, or intersect.

## Goals / Non-Goals

**Goals:**
- Make target body selection visible exactly when the selected operation needs a boolean target.
- Add the missing form fields for basic boolean-capable features without adding feature-specific React branches.
- Preserve explicit `booleanScope` state so preview, commit, history, and rebuild stay deterministic.
- Cover both basic and advanced boolean form behavior in tests.

**Non-Goals:**
- Redesign the feature inspector layout.
- Change kernel boolean execution policy.
- Replace basic feature `booleanScope` with advanced feature participant payloads.
- Add a new form field type or a new dependency.

## Decisions

Keep conditional visibility in feature authoring definitions. The existing form schema already supports `hidden`, field-level errors, reference collections, and picker metadata. Defining target-body visibility in each feature schema keeps business logic next to draft semantics and lets `FeatureInspector` remain a generic renderer.

Use `referenceCollection` for basic feature target bodies. The modeling contract already supports `booleanScope.kind: targetBodies`, and OCC policy documents ordered multi-body boolean handling. A collection field matches advanced features and avoids a later UI/schema change for multi-body joins, cuts, or intersects.

Patch basic target-body fields through `booleanTargetBodyIds`. The shared `toBooleanScope` helper already maps zero, one, or multiple body ids into the correct `FeatureBooleanScope` shape. Operation changes to `newBody` continue to reset scope to `standalone`.

Leave advanced features unchanged except for regression coverage. Sweep, loft, and thicken already hide target bodies for `create` and require at least one target body for `add`, `subtract`, and `intersect`; changing that path would create unnecessary churn.

## Risks / Trade-offs

- [Hidden fields may retain stale form values] -> Mitigate by ensuring `newBody` definitions always use standalone scope and tests assert the contract output after switching back.
- [Basic and advanced operation names differ] -> Mitigate by testing both vocabularies: `newBody`/`join`/`cut`/`intersect` for basic features and `create`/`add`/`subtract`/`intersect` for advanced features.
- [Target selectors could accept invalid target kinds] -> Mitigate by using each feature's existing body-capable selection filter requirement and body-only `referenceCollection` patch handling.

## Migration Plan

1. Add target body collection fields to basic boolean-capable feature form schemas with operation-driven `hidden`, `error`, and `advancedParticipant` metadata.
2. Ensure selection and patch handling preserves target-body ids in `booleanScope` for boolean operations and resets scope for standalone operations.
3. Add focused authoring tests for extrude, revolve, shell, and existing advanced feature conditional visibility.
4. Run `bun run test` and `bun run lint`.
