## Context

The workbench already routes export requests through `ModelingService.exportDocument` into `OpenCascadeKernelAdapter.exportDocument` for geometry formats. That adapter currently validates revision freshness, then returns `occ-export-unavailable` for STL, STEP, and 3MF. The mock adapter returns synthetic payloads so UI and service tests can exercise the modal, but those payloads are not usable CAD exports.

The OCC authoring state already tracks live solid bodies as `OccTrackedBody` records with `TopoDS_Shape` instances. Export should use those live body shapes directly and keep React components unaware of OpenCascade writer details. Existing request/result contracts should remain stable unless implementation exposes a proven missing field.

## Goals / Non-Goals

**Goals:**
- Generate real, non-empty STL, STEP, and 3MF payloads for selected live body targets.
- Keep export generation behind the modeling/kernel adapter boundary.
- Use existing mesh accuracy options for tessellated STL and 3MF output.
- Keep STEP export B-Rep based and free of mesh accuracy behavior.
- Return actionable diagnostics for stale revisions, missing bodies, empty exports, writer failures, and unsupported runtime capabilities.
- Add tests that create simple OCC solids and validate payload signatures, file extensions, MIME types, and target rejection.

**Non-Goals:**
- Importing any exported format.
- Exporting assemblies, all document bodies, or multi-selection targets.
- Changing the export modal UX or adding toolbar export actions.
- Adding cloud/share/export job infrastructure.
- Guaranteeing semantic CAD metadata in 3MF beyond a valid mesh package for the selected body.

## Decisions

### Resolve only live body targets in the OCC adapter

`OpenCascadeKernelAdapter.exportDocument` should accept geometry export only when `request.target.kind === 'body'` and the body id resolves in `runtimeState.authoringState.bodies`. The selected `OccTrackedBody.shape` becomes the export source.

Alternative considered: export from `KernelDocumentSnapshot.render.records`. Render records are presentation meshes, not the authoritative kernel shape, and they cannot produce a proper STEP B-Rep export.

### Split export implementation into small OCC helpers

Add a focused helper module under `src/domain/modeling/occ/` for geometry export. The adapter should remain responsible for request/revision checks and result metadata, while helpers handle shape resolution, meshing, writer calls, temporary filenames, and MEMFS reads.

Alternative considered: implement writer code inline in `opencascade-kernel-adapter.ts`. That file is already large, and inline writer code would make test setup and runtime capability probing harder to isolate.

### Use OpenCascade writers where available, with explicit runtime capability diagnostics

STL should use OpenCascade meshing plus STL writer APIs. STEP should use `STEPControl_Writer` so the output preserves B-Rep geometry. Each writer should write into the OpenCascade.js virtual filesystem, read the generated bytes/text, then remove temporary files.

3MF should first use a native OpenCascade writer if the bundled runtime exposes one. If no reliable 3MF writer is available, implement a minimal 3MF package from tessellated triangles using a direct ZIP dependency and the required 3MF package parts. The fallback must still be behind the OCC export helper and must not change the public export contract.

Alternative considered: keep 3MF unavailable until native OpenCascade support is found. The user specifically asked for actual 3MF export, and a minimal triangle-mesh 3MF is achievable without widening the UI contract.

### Tessellate once per mesh export request

STL and 3MF should run `BRepMesh_IncrementalMesh` on the selected body using `request.options.meshAccuracy.chordTolerance` and `angleToleranceRadians`. Export should verify that triangulated faces yield at least one triangle before returning success.

Alternative considered: reuse viewport render meshes. Viewport mesh generation has picking/rendering concerns and may not preserve all triangles or units needed by export.

### Keep STEP options mapped narrowly

STEP export should honor the selected schema as far as OpenCascade.js exposes `Interface_Static`/writer configuration. Units remain millimeters because the existing contract only permits millimeter. If a schema cannot be configured in the runtime, return an error diagnostic instead of silently ignoring the option.

Alternative considered: always write the default STEP profile. That would make the modal option misleading and weaken testability.

### Treat generated files as payloads, not retained state

Successful exports should return `string` for text formats when practical and `Uint8Array` for binary formats. The service/modal download path already accepts both, so no React state shape should change.

Alternative considered: return `Blob` from the adapter. That would leak browser APIs into domain/kernel tests and make non-browser tests harder.

## Risks / Trade-offs

- [OpenCascade.js filesystem differences] Writer APIs may produce files only through MEMFS paths. Mitigation: centralize temporary path creation, read/remove files in helper functions, and test under the existing Bun/node OpenCascade runtime.
- [3MF fallback complexity] A custom 3MF package can be valid but minimal. Mitigation: constrain it to one mesh object in millimeters, add package-structure tests, and document that richer metadata is out of scope.
- [Large mesh memory use] STL and 3MF exports can allocate large arrays. Mitigation: export one selected body at a time, avoid storing payloads after download, and keep helper allocations local.
- [Schema option support] STEP schema configuration may differ across OpenCascade.js builds. Mitigation: probe required APIs and return `occ-export-writer-unavailable` or equivalent diagnostics when unsupported.
- [Regression from mock tests] Existing mock tests should remain useful but not prove real export. Mitigation: add OCC-specific tests and keep mock tests limited to service/UI behavior.

## Migration Plan

1. Add failing OCC export tests for a simple created solid covering STL, STEP, 3MF, stale revision, and non-body target diagnostics.
2. Add OCC export helpers for body resolution, temporary filesystem output, meshing, and common result metadata.
3. Implement STEP writer support and tests.
4. Implement STL writer support and tests for binary and/or ASCII payload shape.
5. Implement 3MF writer support using native writer if available, otherwise a minimal zipped 3MF mesh package with a direct dependency.
6. Update any existing mock/service tests that imply synthetic geometry export is the final behavior.
7. Run `bun run test` and `bun run lint`.

Rollback is limited to restoring `occ-export-unavailable` in the adapter and removing the helper/tests. The export modal and cadara JSON path do not need rollback for this change.

## Open Questions

- Does the bundled OpenCascade.js runtime expose a native 3MF writer in both browser and node entries, or should implementation go directly to the minimal 3MF package fallback?
- Should STL ASCII be implemented in the first pass if the runtime writer only exposes binary reliably, or should ASCII return a diagnostic until a text serializer is added?
